## Requirements
### Requirement: Employee record model
The system SHALL store an Employee with fields `id` (`EmployeeId`, DB Long sequence),
`fullName` (required), `email` (required, unique, immutable identity key), `role`
(`EmployeeRole`, default `EMPLOYEE`), `jobTitle` (optional), `department` (optional),
`level` (optional enum `JUNIOR`/`INTERMEDIATE`/`SENIOR`), `createdAt`, and `updatedAt`
(server-managed). The entity SHALL NOT carry an `active`/soft-delete flag.

#### Scenario: Required and optional fields
- **WHEN** an Employee is persisted
- **THEN** `fullName`, `email`, and `role` are non-null, `jobTitle`/`department`/`level` may be null, `level` is one of `JUNIOR`, `INTERMEDIATE`, `SENIOR` or null, and `role` is one of `EMPLOYEE`, `ADMIN`

#### Scenario: Timestamps are server-managed
- **WHEN** an Employee is created or updated
- **THEN** `createdAt` is set on create and never changed, `updatedAt` is set on create and refreshed on every update, and neither is accepted from the client

#### Scenario: ID generation
- **WHEN** a new Employee is persisted
- **THEN** its `id` is a Long generated by the database sequence/identity and wrapped in `EmployeeId`

### Requirement: EmployeeContract implementation
The system SHALL provide `EmployeeService` in `employee/service/` that implements the
frozen `EmployeeContract`: `Optional<EmployeeSummary> findById(EmployeeId)`,
`boolean exists(EmployeeId)`, and `Optional<EmployeeSummary> findByEmail(String email)`.
`EmployeeSummary` SHALL carry `id`, `fullName`, `email`, and `role`. Richer fields
(`jobTitle`, `department`, `level`, timestamps) SHALL be exposed only through the
Employee module's own response DTO, not through the contract.

#### Scenario: findById returns summary
- **WHEN** `findById` is called with an existing `EmployeeId`
- **THEN** it returns an `Optional` of `EmployeeSummary` populated with `id`, `fullName`, `email`, and `role`

#### Scenario: findById missing
- **WHEN** `findById` is called with a non-existent `EmployeeId`
- **THEN** it returns an empty `Optional`

#### Scenario: findByEmail returns summary
- **WHEN** `findByEmail` is called with an existing employee's email
- **THEN** it returns an `Optional` of that `EmployeeSummary`; for an unknown email it returns an empty `Optional`

#### Scenario: exists check
- **WHEN** `exists` is called
- **THEN** it returns `true` iff an Employee with that `EmployeeId` is persisted

### Requirement: Self-service employee creation
The system SHALL expose `POST /api/v1/employees` allowing any authenticated user to
create their own profile. The `email` SHALL be bound to the authenticated principal
(`principal.name`); the request body SHALL NOT carry `email` or `role`. The body SHALL
carry `fullName` (required) and optionally `jobTitle`, `department`, and `level`. The
created Employee's `role` SHALL be forced to `EMPLOYEE` (no self-promotion). A successful
create SHALL return the created Employee (unwrapped, camelCase) with HTTP 201. No user
SHALL be able to create a profile for another identity.

#### Scenario: Create own profile with forced EMPLOYEE role
- **WHEN** an authenticated user POSTs `{ fullName, jobTitle?, department?, level? }`
- **THEN** an Employee is created with `email` set to the principal's name and `role` set to `EMPLOYEE`, and HTTP 201 is returned with the created record

#### Scenario: Email and role are derived, not supplied
- **WHEN** the POST body includes an `email` or `role` field
- **THEN** it is ignored or rejected, and the created record's `email` equals the principal's name and `role` is `EMPLOYEE`

### Requirement: Owner-or-admin employee update with admin-only role change
The system SHALL expose `PUT /api/v1/employees/{id}` performing a full replace of
`fullName`, `jobTitle`, `department`, and `level` (omitted optional fields become null),
and of `role` when the caller is an ADMIN. `email` SHALL be immutable: it SHALL NOT be
accepted in the body; a body whose `email` differs from the stored value SHALL be rejected
with 400. The endpoint SHALL be allowed for the record's owner
(`employee.email == principal.name`) or an ADMIN; any other authenticated user SHALL
receive 403. A non-admin caller SHALL NOT be able to change `role`; an attempt to do so
SHALL be rejected with 403.

#### Scenario: Owner updates own profile
- **WHEN** an authenticated non-admin user PUTs to their own record id without changing `role`
- **THEN** the record is updated and returned with HTTP 200

#### Scenario: Admin updates any record including role
- **WHEN** an ADMIN PUTs to any employee id and sets `role`
- **THEN** the record (including `role`) is updated and returned with HTTP 200

#### Scenario: Non-owner non-admin rejected
- **WHEN** an authenticated non-admin user PUTs to another employee's id
- **THEN** the response is 403 with the uniform error envelope

#### Scenario: Non-admin cannot change role
- **WHEN** an authenticated non-admin owner PUTs to their own record with a `role` differing from the stored value
- **THEN** the response is 403 with the uniform error envelope and `role` is unchanged

#### Scenario: Email is immutable
- **WHEN** a PUT body includes an `email` differing from the stored value
- **THEN** the response is 400 with the uniform error envelope

### Requirement: Employee directory listing
The system SHALL expose `GET /api/v1/employees` to any authenticated user, returning the
full directory as `Paged<EmployeeResponse>`. It SHALL accept `offset` (default 0) and
`limit` (default 20, max 100) and `sort=field,direction` where `field` is one of
`fullName`, `email`, `department`, `level`, `createdAt` and `direction` is `asc` or
`desc` (default `createdAt,desc`). No list filtering SHALL be supported in Phase 1.

#### Scenario: Default page
- **WHEN** `GET /api/v1/employees` is called with no parameters
- **THEN** it returns up to 20 employees ordered by `createdAt,desc` with `offset`, `limit`, and `total` in the paged envelope

#### Scenario: Sort whitelist enforced
- **WHEN** `sort` requests a field outside the whitelist
- **THEN** the response is 400 with the uniform error envelope

#### Scenario: Limit clamped
- **WHEN** `limit` exceeds 100
- **THEN** the response uses a limit of 100

### Requirement: Employee fetch by id
The system SHALL expose `GET /api/v1/employees/{id}` to any authenticated user, returning
the Employee (unwrapped, camelCase) with HTTP 200, or 404 with the uniform error envelope
if no Employee has that id.

#### Scenario: Existing employee
- **WHEN** `GET /api/v1/employees/{id}` is called for an existing id
- **THEN** the response is HTTP 200 with the employee record

#### Scenario: Missing employee
- **WHEN** `GET /api/v1/employees/{id}` is called for a non-existent id
- **THEN** the response is 404 with the uniform error envelope

### Requirement: Email uniqueness
The system SHALL enforce that `email` is unique across Employees. A create or the
identity-binding of a principal whose email already exists SHALL return 409 Conflict
with the uniform error envelope.

#### Scenario: Duplicate email on create
- **WHEN** a principal whose email already maps to an Employee attempts to POST
- **THEN** the response is 409 with the uniform error envelope and no second record is created

### Requirement: Employee validation
The system SHALL validate Employee input: `fullName` non-blank with a max length; `email`
non-blank and format-valid (only when supplied by the system on create); `level` either
null or one of `JUNIOR`/`INTERMEDIATE`/`SENIOR`; `jobTitle` and `department` free text
with a max length; `role`, when supplied on PUT by an admin, one of `EMPLOYEE`/`ADMIN`.
Any violation SHALL return 400 with the uniform error envelope.

#### Scenario: Invalid level
- **WHEN** a POST or PUT supplies `level` not in the enum
- **THEN** the response is 400 with the uniform error envelope

#### Scenario: Blank full name
- **WHEN** a POST or PUT supplies a blank `fullName`
- **THEN** the response is 400 with the uniform error envelope

#### Scenario: Invalid role on admin PUT
- **WHEN** an admin PUT supplies a `role` not in `EmployeeRole`
- **THEN** the response is 400 with the uniform error envelope

### Requirement: Employee Liquibase table and admin seed
The system SHALL add a Liquibase changeset under `db/changelog/modules/employee/`
creating the `employee` table (columns matching the record model including `role`, a
unique constraint on `email`, and an identity/sequence for `id`) and seeding exactly one
`admin` Employee with email `admin@staff.eng` and `role = ADMIN`. No other employees
SHALL be seeded.

#### Scenario: Table and seed applied
- **WHEN** Liquibase runs
- **THEN** the `employee` table exists with a unique `email` constraint, a `role` column, and exactly one seeded `admin@staff.eng` row with `role = ADMIN`

### Requirement: Frontend employee feature
The system SHALL provide `frontend/.../features/employee/**` with an `EmployeeList`
component, an `EmployeeDetail` component, and an `EmployeeStateService` using Angular
Signals (in-memory only). A lazy route for the feature SHALL be appended as exactly one
line in `routes.ts`. The create form SHALL omit the email and role fields (derived
server-side). Edit affordances SHALL be gated by RBAC: an ADMIN can edit any record and
may change role; a non-admin sees edit only on their own record and cannot change role.
The list SHALL be visible to all authenticated users.

#### Scenario: Lazy route loads
- **WHEN** the employee route is navigated to
- **THEN** the feature loads on demand via `loadChildren`

#### Scenario: Create form has no email or role field
- **WHEN** an authenticated user opens the create form
- **THEN** the form collects `fullName`, `jobTitle`, `department`, and `level` only; email and role are not entered

#### Scenario: RBAC gates edit and role affordance
- **WHEN** a non-admin views another employee's detail
- **THEN** no edit affordance is shown; an admin viewing any detail sees the edit affordance and a role control
