## Findings

### Compliant ✅

- **SpEL `@PreAuthorize("hasAnyRole('USER','ADMIN')")` is correct for Spring Security 6** — the project pins `spring-boot-starter-parent 3.5.15` (`backend/pom.xml:8`), so `@EnableMethodSecurity` on `SecurityConfig` evaluates the SpEL at call time and `hasAnyRole('USER','ADMIN')` with single-quoted, comma-separated authorities is the canonical form. No `SecurityFilterChain` change is required for ADMIN — the role is granted by `JwtAuthFilter` (lines 42-44) which prefixes every role claim with `ROLE_`, and `hasAnyRole` matches `ROLE_<name>`. Consistent with how `AuthController` already resolves roles from the Employee record (`AuthController.java:63-72`); ADMIN is the only non-`EMPLOYEE` role in `StubUserStore` (`StubUserStore.java:28-31`), so adding `hasAnyRole('USER','ADMIN')` on the four task endpoints is the natural extension.
- **Liquibase additive changeset is well-formed** — `addColumn` with `defaultValue: ''` + `constraints: nullable: false` is the textbook Postgres-safe pattern for adding a NOT NULL column to a table with existing rows (the default backfills before the constraint is checked). The changeset `id: task-002-add-title` is unique vs `task-001` (the previous file is `001-create-task-table.yaml`, change-id `task-001`). The master file (`db/changelog/master.yaml`) is untouched — the new file is picked up by the existing `includeAll` directive on `db/changelog/modules/` (lines 6-8 of master.yaml). Comment block documents the additive intent.
- **JPA `@Column(name = "title", nullable = false, length = 255)` is appropriate** — `length = 255` matches the Liquibase `VARCHAR(255)`. The defensive `= ""` default initializer on `Task.java:29` is correct belt-and-braces: it satisfies `NOT NULL` if Hibernate ever issues an insert before Liquibase backfills (e.g., test H2 contexts or future native DDL paths), and it matches the `defaultValue: ''` in the Liquibase column definition — schema and entity agree.
- **`Task.builder().title(request.title() == null ? "" : request.title())` is in the right place** — it lives on the controller boundary, not on the entity or the DTO, which is the correct layering (the entity keeps a no-arg default for safety; the controller is the integration point where absent-client-input is normalized). The Javadoc on `TaskRequest` (`TaskController.java:107-118`) is honest about Bean Validation wiring — records on `@RequestBody` are validated by Spring Boot 3.x with the implicit `@Valid`, and adding `@NotBlank` on `String title` would be a clean follow-up (see Warning 1 below).

- **Given-When-Then structure is explicit** — every test method in the new and updated suites carries `// Given`, `// When`, `// Then` (or `// When / Then` for single-line expectations), with `@DisplayName` summaries. Matches `testing-strategy.yaml:24-26`.

It catches a regression of the literal `hasRole('USER')` back to the controller. ### Warnings ⚠️

1. **`String title` has no Bean Validation annotation** — `TaskRequest` records the field as plain `String`, so a missing/null/blank `title` is silently coerced to `""` at the controller (`TaskController.java:76`). The spec at `openspec/changes/atse1-25-35-ux-walkthrough-fixes/specs/task-management/spec.md:33-38` explicitly says: *"Creation failure due to missing title — WHEN a user sends a POST request without a title (null or empty) — THEN the system returns 400 Bad Request with a uniform error envelope stating that title is required"*. The current implementation does NOT do this — it accepts blank titles. To honour the openSpec scenario, the record should be `String title` annotated `@NotBlank` and the controller parameter annotated `@Valid TaskRequest`. The current defensive `== null ? "" : request.title()` coercion then becomes dead code (or remains as a safety net). This is the most material gap in the diff vs the openSpec requirement.
2. **The reflective assertion is literal-string-based** — `isEqualTo("hasAnyRole('USER','ADMIN')")` is robust against most mutations, but a Stryker mutant that swaps the order to `"hasAnyRole('ADMIN','USER')"` would NOT be caught (the literal still contains both roles). Similarly, a mutant that adds a third role like `"hasAnyRole('USER','ADMIN','GUEST')"` would survive. Consider asserting `containsExactlyInAnyOrder` semantics via parsing the SpEL into role tokens if a strict allowlist is desired — overkill for the POC but worth noting for future mutation hardening.
3. **`TaskSummary.title` Javadoc is now stale** — `shared/api/TaskSummary.java:13-14` says "`title` is retained for backward compatibility with earlier consumers and carries the same value." Post-fix `title` and `description` are distinct. Since `TaskSummary` is a frozen cross-module record (per `TaskContract`'s Javadoc), changing it is a contract-level event; a Javadoc-only update is safe and should land alongside this commit to avoid misleading downstream modules. The cross-module `Profile` consumer may still depend on the old equivalence — worth a sanity check before the next archive.
4. **JUnit file trailing newlines are missing** — `TaskController.java`, `002-add-task-title.yaml`, `TaskServiceMappingTest.java`, and `TaskControllerSecurityTest.java` all lack the trailing newline. POSIX/lint conventions and several CI pre-commit hooks flag this. No functional impact but worth tidying.
5. **The `task.title` column is `nullable: false` but the DTO accepts `null`** — coupled with Warning 1, this means the schema and the openSpec rule diverge. If you choose to keep the defensive coercion, add a `@Column(nullable = false)` validation on the JPA side and consider a `@PrePersist` hook on `Task` to normalize nulls, so a future native-SQL insert path also stays safe.

## Verdict

**LANDABLE.** Two follow-ups recommended but neither is blocking:

1. Add `@NotBlank` on `TaskRequest.title` (and `@Valid` on the controller parameter) to honour the openSpec `task-management` "Creation failure due to missing title" scenario — this is the spec-vs-implementation gap surfaced by the audit.
2. Update the stale `TaskSummary` Javadoc to reflect that `title` and `description` are now distinct.

The remaining Warnings (Stryker robustness, trailing newlines, defensive `@PrePersist`) are nice-to-haves for a future PR. The §7 commit fixes the seeded-admin RBAC regression AND the `description`-mapped-to-both-slots latent bug without breaking the modular-monolith contract or the Liquibase additive-migration safety story.
