## Warnings ⚠️

- **api-standards.yaml -> responses.null_handling ("Exclude Nulls") — `note` may be set to literal `null` on PATCH.** `UpdateInteractionRequest.note` is a plain `String` with no validation (UpdateInteractionRequest.java:21). The service writes `entity.setNote(note)` unconditionally (InteractionService.java:139) — a PATCH with `{"type":"check-in","note":null}` is accepted and clears the note. The `Interaction.note` column is `columnDefinition = "text"` (Interaction.java:59) and not `nullable = false`, so the DB allows it. The MISSION model treats the note as optional free-text, and "clearing" a note on edit is arguably the user's intent — but the constitution's "Exclude Nulls" rule is normally interpreted at the JSON-serialization layer (drop nulls when responding), not at the request layer. A PATCH that explicitly sends `null` is a valid client signal. **Remediation 🛠️:** none required by the constitution. Optional UX hardening: add a `@Size(max = N)` to `UpdateInteractionRequest.note` to guard against unbounded note bodies, and document the PATCH semantics (`null` clears, absent does not) in the DTO JSDoc so the wire contract is explicit.

- **backend-architecture.yaml -> persistence_strategy.transaction_model — `update` is not `@Transactional`.** `InteractionService` has no class-level `@Transactional`, and the new `update` method has no method-level annotation. The path is `findById` → set fields → `save`, which is a single aggregate write that does not need an explicit transaction in the simple case (Spring Data JPA opens an implicit transaction per repository call). However, if the `update` path is ever extended (e.g. to bump a `Task` count, write an audit row, or fan out a notification), the lack of an explicit transaction boundary will become a correctness bug. **Remediation 🛠️:** none required for §4. The audit flags this as a future-proofing concern, not a §4 regression. A coordination PR could add a class-level `@Transactional` on `InteractionService` to match the spirit of `backend-architecture.yaml -> persistence_strategy.transaction_model` ("Standard Spring @Transactional").

- **ROADMAP §2.7 (additive carve-out claim) — `verifyEditable` shape on the contract.** The frozen `InteractionContract.verifyEditable` (InteractionContract.java:34-37) returns `Optional<InteractionId>`. The §4 service override matches the signature exactly. However, the carve-out row in `ROADMAP §2.7` is the *only* documentation of the additive method — the parameter list and return type live in the interface Javadoc but are not separately called out in the roadmap table. A future reader scanning `ROADMAP §2.7` for "what does the additive `verifyEditable` return" has to chase the `InteractionContract.java` source. **Remediation 🛠️:** one-line ROADMAP amendment appending the parameter and return shape inline. Optional; the source of truth is the interface.

- **`InteractionAccessControlTest` does not cover the new `update` endpoint.** The pre-existing test file (InteractionAccessControlTest.java) reflects on `create`, `listBySubject`, and `getById` (lines 25-57) and asserts the `@PreAuthorize("hasRole('ADMIN')")` declaration on each. The new `update` method (controller line 87) has the same `@PreAuthorize` declaration (controller line 88), but the file does not assert it. A reflection test for the new method would close the RBAC unit-test gap. **Remediation 🛠️:** add a 4th test `updateEndpointRequiresManagerRole` to `InteractionAccessControlTest` reflecting on `InteractionController.class.getMethod("update", Long.class, UpdateInteractionRequest.class, UserDetails.class)` and asserting `@PreAuthorize` with value `"hasRole('ADMIN')"`. This is a 1-test addition that closes the RBAC declaration gap end-to-end.

- **api-standards.yaml -> security.authentication (Bearer JWT) — username-to-EmployeeId binding is a code smell flagged in the controller JSDoc.** The controller (lines 92-99) parses the principal username with `Long.parseLong(userDetails.getUsername())`. This is documented as a Phase 0 limitation (the principal is a username with no `EmployeeId` mapping), and the test `updateBindsActorIdFromPrincipalUsername` (line 224) pins the behaviour. The controller JSDoc (lines 33-35) honestly records the deferred design ("D3/D7"). **Remediation 🛠️:** none required for §4. The code is correct *as documented* and the openSpec is the right place to track the deferred mapping. A coordination PR is needed when the principal shape evolves.

- **testing-strategy.yaml -> backend.mutation_testing (PITest) — one unexercised branch in `update`.** The PITest-recommended way to ensure all branches are covered is to assert on the no-op path for `isAdmin && !match` (line 131 with `isAdmin=true`). The current `updateChangesTypeAndNoteForAdminWithoutTouchingAuditFields` test (line 232) does exercise this — it uses `isAdmin=true` with `actor=99L` against `facilitator=2L` and expects the update to proceed. A Stryker-equivalent test for the service confirms the admin branch is taken. The audit verifies this is covered; the warning is that a future maintainer who deletes the admin test would lose coverage of the admin-no-match path. **Remediation 🛠️:** no change today. PITest CI will surface the regression if the test is ever dropped.

- **Cross-feature consistency — the controller method `update` reads `isAdmin` from the principal's authorities (line 97-98) and passes it to the service.** The service then re-implements the same admin check (line 131). This is the documented pattern (controller-level coarse RBAC + service-level fine-grained ownership), but it means the same boolean is computed/forwarded twice. A future refactor could centralize "is this principal an admin" in a shared helper. **Remediation 🛠️:** none required. The pattern is consistent with the rest of the codebase and is the right separation of concerns for a POC.

- **JSDoc on `InteractionService.update` uses `@throws` for the 404 path but the actual exception is also thrown on the missing-id path.** The JSDoc (lines 124-126) correctly documents the dual meaning of the 404, but a future reader may miss the collapse and treat the two paths as a single throw. **Remediation 🛠️:** the JSDoc is correct. Optional: extract the collapse logic into a private `assertEditable(Interaction, EmployeeId, boolean)` method that throws on failure, and have `update` call it before the null-type check — makes the intent explicit and gives a single place to mutate the RBAC policy. Pure refactor, no behaviour change.
