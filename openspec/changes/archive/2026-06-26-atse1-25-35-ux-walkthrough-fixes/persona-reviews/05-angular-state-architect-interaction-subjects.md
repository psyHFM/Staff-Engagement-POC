# §5 (ATSE1-33) Angular State Architect Review

## Summary
**PASS** — `availableSubjects` was correctly swapped from a hardcoded 3-entry stub into a signal that is hydrated by `GET /api/v1/employees` via the existing state-service side-effect channel, with the public surface (`subjects` computed) preserved so neither consumer had to change its read path.

## Findings

### Compliant ✅
- **Signal-based state, no legacy `BehaviorSubject`** — `frontend/src/app/features/interaction/interaction-state.service.ts:45` declares `private readonly availableSubjects = signal<EmployeeOption[]>([])`. Grep across `frontend/src/app/features/interaction` finds zero `BehaviorSubject`/`Subject<` usages (the service imports only `Observable`, `finalize`, `tap` from `rxjs`, never `BehaviorSubject`).
- **Public read API stays a `computed()`** — `interaction-state.service.ts:47` exposes `readonly subjects = computed(() => this.availableSubjects())`. This is the required derived-state pattern (frontend-state.yaml → `derived_state.mechanism: "Pure computed()"`) and avoids the consumer ever touching the private writable signal.
- **Side-effect lives in the state service handler** — `loadSubjects()` (interaction-state.service.ts:54-71) is the single entry point. It uses `catchApiError()`, `finalize(() => this.endLoad())`, and a `tap({ next, error })` pair to bridge the RxJS HTTP stream into a signal write. `beginLoad()`/`endLoad()` (inherited from `StateService`) drives `loading` symmetrically, so `isLoading()` returns `false` on both success and failure — no `finalize` race.
- **Component-side read path unchanged** — `interaction-page.html:10,12,38` still binds to `state.subjects()` (the computed). The dropdown `(change)` handler in `interaction-page.ts:55-59` still calls `state.selectSubject(...)` and `state.loadHistory()`, never the private signal. The pre-selection in `ngOnInit()` (`interaction-page.ts:45-52`) uses the public `subjects()` computed — so once `loadSubjects()` resolves the dropdown auto-fills. This is a clean example of unidirectional flow: `Component Action → State Service → API → Signal Update → UI Update`.
- **Error contract routes through the same error signal** — `interaction-state.service.ts:67` writes `lastError.set(err)` on failure; `error` is exposed via `computed()` (line 75) and the page template (`interaction-page.html:19-23`) already reads it. No new error surface introduced.
- **Pagination shape matches the backend `Paged<T>` contract** — `loadSubjects()` requests `{ offset: 0, limit: 100 }` and reads `page.content` (interaction-state.service.ts:58, 64). This mirrors `loadHistory()` (line 92-105), keeping the async pipeline consistent with the rest of the service. `EmployeeResponse` (from `frontend/src/app/features/employee/employee.types.ts:62-72`) is correctly imported as the DTO and projected to `EmployeeOption` (dropping `email`, `role`, etc.) — keeps the consumer's contract narrower than the wire shape.
- **Persistence policy respected** — No `localStorage` / `sessionStorage` is touched; the list is in-memory and resets on reload. The JWT carve-out (frontend-state.yaml → `persistence.carve_outs`) remains the only persistence point and is unaffected.
- **Component uses `ChangeDetectionStrategy.OnPush`** — `interaction-page.ts:34` and `log-interaction.ts:19` both declare `OnPush`. Combined with Signals this removes the need for manual change detection anywhere in the consumer chain.
- **Tests assert behaviour, not stub identity** — `interaction-state.service.spec.ts:67-108` now verifies: empty initial state (`expect(service.subjects()).toEqual([])`), the exact API call (`expect(apiClientSpy.get).toHaveBeenCalledWith('employees', { offset: 0, limit: 100 })`), the projected shape (`toEqual([{ id, fullName }, ...]`), and the error branch keeping `subjects` empty while populating `error` and clearing `isLoading`. These are pure BDD `Given / When / Then` cases.
- **`toEqual()` over `toBe()` for objects** — the new specs use `toEqual(...)` for all object/array assertions (e.g. `interaction-state.service.spec.ts:69, 91-94, 105-106`); `not.toBeNull()` is used for the single null-sentinel case (`:250`). `toBeTruthy()` is avoided.
- **No `.set()` racing `.update()`** — the service uses `.set()` consistently for full-replacement writes (`availableSubjects`, `interactions`, `lastError`, `lastCreated`, `selectedSubject`) and never mixes `.update()` onto the same signal. No microtask interleaving concern.

### Warnings ⚠️
- **W1 — `loadSubjects()` request page-size is a magic number, not derived from the directory's sort whitelist** (severity: low; `interaction-state.service.ts:58`). The hard-coded `limit: 100` will silently truncate when more than 100 employees are seeded. *Suggested fix (out-of-scope for §5, but flag for §7+)*: iterate via `offset`/`limit` or use a server-side "all" parameter; at minimum extract `SUBJECTS_PAGE_SIZE` to a named constant and assert on it in the spec rather than the literal.
- **W2 — `availableSubjects` is private but the comment hints at external coordination** (severity: low; `interaction-state.service.ts:40-44`). The JSDoc says "the seeded admin and employee rows live at ids 1 and 2 so the previously-stubbed values continue to work" — that invariant lives in `defaultFacilitator()` (line 198-211) as well. *Suggested fix*: add an `interaction-state.service.spec.ts` case that asserts `defaultFacilitator()` resolves ids that are present in the loaded `subjects()` list, so the two pieces are coupled by a test rather than by JSDoc prose.
- **W3 — `selectSubject` is called on a synchronous read of an empty `subjects()` signal** (severity: low; `interaction-page.ts:48-52`). The first `ngOnInit` run will find `subjects()[0]` undefined because the HTTP call hasn't completed yet, so the pre-selection branch is skipped on first paint. Today this is benign (the user manually picks), but the comment "Pre-select the first stub subject so the page is not empty on arrival" reads as if it should still happen — it does not, because the stub is gone. *Suggested fix (future)*: move the pre-select inside the `tap` callback in `loadSubjects()` (or chain a `.subscribe` that calls `selectSubject` and `loadHistory` after the page resolves). The new spec already covers the empty-state behaviour, so this is purely a UX nit.
- **W4 — Spec does not cover a second `loadSubjects()` call superseding the first** (severity: low; `interaction-state.service.spec.ts:72-95`). If a component calls `loadSubjects()` twice, the latest response wins (correct), but the loading flag's begin/end pair on overlapping calls could in principle flicker `isLoading` to `false` mid-flight of the earlier call. The current implementation is correct because RxJS `subscribe()` is sequential per subscription, but a regression here would be silent. *Suggested fix*: add a "second call replaces first" spec that swaps the mock between the two `apiClientSpy.get.mockReturnValue(...)` calls.
- **W5 — No cancellation / no `takeUntilDestroyed` on `loadSubjects()`** (severity: low; `interaction-state.service.ts:70`). The existing `loadHistory()`/`createInteraction()`/`updateInteraction()` share the same pattern, so this is a pre-existing stylistic choice, not a §5 regression. *Suggested fix*: consider exposing `Observable<Paged<EmployeeResponse>>` and converting to a signal via `toSignal()` with `inject(DestroyRef)` for consumer-driven lifetime — but again, that would touch §3-§4 parity.

### Violations ❌
None.

## Verdict
**LANDABLE** — go for §5. The swap is the minimum change required: same public computed, same handler signature, side effect still co-located with the state it updates, error routed through the same envelope. The five warnings above are quality-of-life items for the next iteration, not gates for this PR.
